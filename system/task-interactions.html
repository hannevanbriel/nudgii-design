Task Interactions

The complete spec for how users interact with tasks: Done, Skip, and Snooze. Covers behavior, visual state, copy, confirmation logic, and the task instance model. This is the single source of truth for engineering implementation.

3 actions: Done, Skip, Snooze Frequency-aware logic Task instance model Confirmation rules Consistent across all screens
Consistency principle
Same-looking elements must produce identical interactions everywhere. A task card on Today, Plan, and S-17 item detail behaves exactly the same way. No screen-specific exceptions.
Task instance model
Active + projected instances
Each recurring task has exactly one active instance (the next due date, actionable). The Plan tab renders projected future instances as ghost items within the visible time window. Ghosts are calculated on render, not stored in the database. Only the active instance can be acted on (Done/Skip/Snooze).
ConceptDescriptionDatabase
Active instance
The "real" task
 The next occurrence of a recurring task. This is what the user sees and acts on. Has next_due_at, last_completed_at, last_skipped_at, and all action state. tasks table row with next_due_at
Projected instance
Ghost / preview
 A future occurrence calculated from the active instance's due date + interval. Shown on Plan tab and S-17 item detail as a lighter, non-actionable preview. Helps users see their maintenance rhythm over time. Not stored. Calculated: active.next_due_at + (N * interval)
Recurrence pattern Stored on the task template, not on instances. Defines interval (weekly, monthly, every 6 months, yearly, seasonal). One-time tasks have no recurrence pattern. tasks.interval (e.g., P1M, P6M, P1Y)
Where each type shows
Today tab: Active instances only (due now, this week, coming up). No ghosts. This is the "what needs my attention" view.

Plan tab: Active instances + projected ghosts for visible months. Ghosts render at 40% opacity, no checkbox, no chevron, no action buttons. This is the "what does my year look like" view.

S-17 item detail: Active instance (full actions) + next projected instance below it as a preview card. Shows the rhythm for that specific item.

My Stuff: Status badges based on active instance only. "1 task waiting" counts active instances, not projections.
Projected ghost visual treatment
Opacity: 40% on entire row
Checkbox: Hidden (not actionable)
Chevron: Hidden (not expandable)
Action buttons: None
Badge: Date only, no status color
On tap: Tooltip: "This is a projected future occurrence based on your [interval] schedule. It becomes active after you complete or skip the current one."

Label (optional): Small "projected" tag in hint color if needed for clarity.

Cards never collapse, move, or disappear on action
When the user taps Done, Skip, or Snooze, the card stays in place. It gets a visual state change (muted, strikethrough, check) but does not animate away, collapse, or move to another section. The user should see the result of their action in context. Movement to "Done today" section happens at end of day, not immediately.

Skip: detailed spec
Skip never means "skip forever"
"Skip" always means "not now." It postpones the task, it never removes it. For non-recurring tasks, "skip" is effectively "remove" and requires explicit confirmation. The word "skip" must always be paired with the return date so the user understands the consequence.

Skip push logic: how far forward?

Task intervalSkip pushes byRationaleExample
Weekly
7 days
 +1 week (full interval) Natural rhythm. Task comes back at same cadence. Due Mon Apr 7 → skipped → due Mon Apr 14
Bi-weekly
14 days
 +2 weeks (full interval) Same rhythm, next cycle. Due Apr 7 → skipped → due Apr 21
Monthly
~30 days
 +1 month (full interval) Natural monthly cadence. Due Apr 4 → skipped → due May 4
Every 2-3 months
60-90 days
 +1 month Full interval is too long. User is likely busy this week, not abandoning for 3 months. One month is a reasonable postponement. Due Apr 4 (every 3mo) → skipped → due May 4
Every 4-6 months
120-180 days
 +1 month Skipping a 6-month task for another 6 months is dangerous (warranties, safety). One month gives breathing room without losing the task. Due Apr 4 (every 6mo) → skipped → due May 4
Yearly
365 days
 +1 month Annual tasks are high-stakes (boiler service, car inspection). Pushing by a year could void warranties or create safety risks. Due Apr 4 (yearly) → skipped → due May 4
Seasonal
Variable window
 +1 month Seasonal tasks have a window (e.g., "prune in March-April"). Skip within window = +1 month. If window closes, task moves to next season's window. Due Mar 15 (seasonal spring) → skipped → due Apr 15
Non-recurring
One-time
 Removed from list There's no "next time." Skip = remove. Requires confirmation. "Replace smoke detector battery" → skipped → gone (with confirm)
The rule in one sentence
Interval ≤ 1 month: skip pushes by the full interval.
Interval > 1 month: skip pushes by 1 month.

This means "skip" always means "not now, try again soon" rather than "forget about it for the full cycle."

Skip button: always shows the consequence

The Skip button is never just "Skip." It always includes the return date so the user sees what will happen before they tap.

Confirmation rules

IntervalOn tapWhy
≤ 1 month Direct skip. No confirmation. Toast with Undo (5s). Low stakes. Task returns quickly. Undo covers mistakes.
2-3 months Direct skip. No confirmation. Toast with Undo (5s). Medium stakes but +1 month is reasonable. Undo covers mistakes.
4+ months Confirmation required. Bottom sheet explaining consequence, then toast + Undo. High-stakes tasks. User must understand what "skip" means for a yearly service vs. a weekly check.
Non-recurring Confirmation required. Bottom sheet: "This is a one-time task. Skipping removes it from your list." Irreversible (no next occurrence). Must be explicit.

Snooze: detailed spec
Snooze vs Skip
Snooze = "remind me later today/this week." The task stays active, just the reminder timing shifts. No impact on next_due_at or interval calculations.

Skip = "not doing this one, move to next occurrence." The active instance advances. Skip affects the maintenance schedule.

Screen-specific implementation notes
Today tab (S-09)
The daily action view
Shows: Active instances only (due today, this week, coming up). No projected ghosts.

Section counts: Update dynamically when tasks are done/skipped. "3 things need attention today" recalculates. When all done: "All done for today. Nice work."

Done section: "Done today" appears at bottom when any task is completed. Resets at midnight.

Snoozed cards: Stay in their current section with muted state. Badge updates to snooze time.
Plan tab (S-09)
The timeline view
Shows: Active instances + projected ghosts for visible months.

Ghost items: 40% opacity, no checkbox, no chevron, no actions. Tappable for tooltip only.

Month ratios: Update on done/skip. Ratio counts active instances only, not ghosts.

Snoozed cards: Stay in month with muted state. Date column updates to show snoozed-to date.
S-17 Item detail
Per-item task list
Shows: Active instance for each task (full actions) + one projected ghost below each (preview of next occurrence).

Task cards: Expand to show "Why it matters" + "How to complete" + action row (Skip/Snooze/Done).

All actions behave identically to Today and Plan tab versions. No screen-specific exceptions.

Skip confirmation: Uses same bottom sheet pattern as Today/Plan.
Edge cases
ScenarioBehavior
Snooze then Done
User snoozes, then taps checkbox
 Snooze clears. Task marked done. snoozed_until = null, last_completed_at = now. Normal done flow.
Snooze then Skip
User snoozes, then later skips
 Snooze clears. Normal skip flow. snoozed_until = null, skip logic applies.
Skip a snoozed task
Task was snoozed, user expands and skips
 Same as above. Snooze state clears, skip takes over.
Undo after Undo
User undoes, then re-does the action
 Each undo fully restores. The user can then re-tap Done/Skip/Snooze. No "undo the undo" button.
Multiple quick actions
User does 3 tasks fast in sequence
 Each action replaces the previous toast. Only the most recent action can be undone via toast. Earlier actions are committed.
Offline skip/done Action applies locally immediately. Syncs to server when connection returns. Conflict resolution: last write wins.
Seasonal task outside window
User tries to skip a seasonal task after its window closes
 Skip moves to next season's window start date, not +1 month. E.g., "Prune olive tree" (spring) skipped in May → next due Feb next year.
