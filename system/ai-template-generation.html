AI-generated item templates

When a user adds an item that doesn't exist in the curated template library, nudgii uses the Claude API to generate a complete maintenance template on the fly. The user reviews the AI-generated tasks before accepting, and popular templates are promoted to the curated library over time.

Spec Engineering handoff v1.0 · April 2026

How it works

1
User adds an unknown item
Via Browse (search, no template match), Scan (Claude Vision identifies type but no curated template), or Tell me (voice/text describes something we don't have).
↓
2
Category assignment
If not obvious from context, nudgii asks: "What kind of thing is this?" with category chips (Home, Vehicle, Garden, Subscriptions). If from scan, Claude Vision usually infers the category.
↓
3
Claude API: generate template
Single API call to claude-sonnet-4-6. Input: item name, category, brand/model (if known), country. Output: structured JSON with tasks, why/how content, personality name, detail fields, S-17 variant mapping.
↓
4
User reviews on Add Item Review
Same screen as curated templates. "AI-generated" badge visible. User can edit task names, remove tasks, adjust cadence, add custom tasks. All changes are per-item, not global.
↓
5
Item created with AI template
Tasks saved. Source marked as "AI-generated." Template data stored for the learning loop. User can edit everything on S-17 later.

API call: prompt design

System prompt (fixed, stored server-side)

You are nudgii's maintenance knowledge engine. Given an item that a homeowner owns, generate a practical maintenance schedule. Rules: - Generate 2-5 tasks (not more). Focus on what actually prevents problems. - Each task needs: name, cadence, whether it's DIY or needs a professional, typical cost. - Write "Why it matters" in warm, factual language. Explain the consequence of skipping, not the guilt. - Write "How to do it" as 2-4 numbered steps. Practical, specific, no jargon. - Suggest a personality name: short (1-2 words), affectionate, obvious. - If the item has consumables (filters, salt, oil), flag them with product name and typical cost. - Adapt to the user's country for regulations, seasons, and pricing. - Use time windows, not deadlines. "Every 6-8 weeks" not "due on March 15." - Map to an S-17 variant: home_appliance, small_appliance, vehicle, plant, garden_tool, subscription. Respond with valid JSON only. No markdown, no explanation.

Sub-types (Car, Boiler, Oven, Garden pond)

Curated items with subtype_required: true ask the user a sub-type question before tasks generate. For AI-generated items, the AI must decide whether the item needs a sub-type AND either identify it from available context or flag that we need to ask.

Extended AI response schema (sub-type aware)

{ "item_type": "car", "s17_variant": "vehicle", "subtype_required": true, "subtype_question": "What does it run on?", "subtypes": ["petrol", "diesel", "electric", "hybrid", "lpg"], "subtype_detected": "electric", // from scan/conversation, or null if unknown "subtype_confidence": 0.92, // 0-1, low confidence = show picker without pre-selection "tasks": [ { "name": "MOT inspection", "applies_to": "all", ... }, { "name": "Oil change", "applies_to": ["petrol", "diesel", "hybrid", "lpg"], ... }, { "name": "HV battery health check", "applies_to": ["electric", "hybrid"], ... } ] }

AI detection rules

Sub-type confidence tiers

  • High (>0.85): AI pre-selects sub-type in the chip picker. User can override with one tap.
  • Medium (0.50-0.85): AI shows the chip picker with its guess highlighted but not selected. "Looks like an EV — is that right?"
  • Low (<0.50): Chip picker appears empty. User picks manually.
  • Null (no data): Same as low confidence. Normal user-picks flow.

Signals Claude Vision uses for sub-type detection

  • Car fuel: charging port = electric. Fuel cap + "diesel" label on cap = diesel. LPG filler cap (secondary cap) = LPG. Dual ports = hybrid. Tesla-like vehicle = electric.
  • Boiler type: info plate text. "Vaillant ecoTEC" = gas combi. "Viessmann Vitotronic" = varies. Plate reading is the most reliable.
  • Oven: visible burners = gas. Just elements/no flame = electric.
  • Garden pond: visible fish = fish pond. No fish visible for 10+ seconds = decorative (with low confidence flag because fish may hide).

When the AI identifies an unlisted item that needs sub-type

Example: user says "Tell me about my Aga" (a specific brand of cast-iron cooker). This is not in the curated library. AI recognizes it's cooker-like, but:

  • AI determines sub-type IS relevant (gas vs electric vs oil Aga exist)
  • AI sets subtype_required: true in the response
  • AI provides the sub-type options based on what's plausible for that item type
  • User still sees the chip picker on Add Item Review
  • Tasks are filtered by their applies_to arrays once user picks

This means the sub-type pattern is not limited to the 4 curated items — it's a capability AI can invoke for any item where sub-type matters.

UX on Add Item Review

AI-generated badge

When tasks are AI-generated (not from curated library), the Add Item Review shows a subtle badge below the task preview header: "Tasks suggested by AI" in hint color with a sparkle icon. This sets expectations: tasks are a starting point, not a locked schedule.

Edit affordances

Each AI-generated task card has: an X button to remove, a tap-to-edit title, a cadence dropdown to adjust frequency. "Add custom task" ghost link below the list. These affordances are slightly more prominent than on curated templates, because AI tasks need more user validation.

Brand-specific content

When brand + model are known (from scan), the AI tailors steps to the specific product. "Check your De'Longhi Magnifica S manual" instead of "check your manual." The specificity builds trust. If brand is unknown, steps are generic but still practical.

Loading state

API call takes 1-3 seconds. Show: category wash hero with item name (immediate), "nudgii is figuring out what this needs..." shimmer placeholder where task cards will appear, ii mark pulse animation. When tasks load, they appear with a staggered fade-in (40ms delay per card).

Learning loop: AI to curated

AI-generated templates improve over time. The system tracks which AI templates get accepted, edited, or rejected, and uses this data to promote popular templates to the curated library.

1
Track AI template usage
For each AI-generated item: store the original AI response, user edits (added/removed/renamed tasks), and acceptance rate. Source field: "ai_generated".
↓
2
Cluster similar items
Group AI-generated items by normalized item_type. "Espresso machine," "coffee maker," "Nespresso" might cluster into one group. Track which tasks are kept most often.
↓
3
Promotion threshold
When an item type has 20+ AI-generated instances with >80% task acceptance rate, flag it for manual review. A human (Hanne) reviews the most common task set and decides whether to promote to curated.
↓
4
Curated template created
Item type added to item-templates.html. Tasks refined based on user editing patterns. Future users get the curated template instead of an AI call. Faster, more consistent, no API cost.

The flywheel

More users → more AI-generated templates → more data on what tasks people keep → better curated templates → less AI needed → lower cost + faster experience. The AI bootstraps the knowledge, users validate it, curated templates capture it permanently.

Scan integration

AI template generation works seamlessly with the existing AI scan flow:

Scan + template in one call

When the user scans an item that doesn't match a curated template, the Claude Vision call already returns item_type, brand, model. Instead of two API calls (scan + template), combine into one: the scan system prompt includes the template generation instructions. One call returns both identification AND maintenance tasks. Cost: ~$0.01 (vision input is the photo).

Scan + curated template

When the scan identifies an item that DOES match a curated template (e.g., scans a Vaillant boiler label), skip the AI generation entirely. Use the curated template. Faster, cheaper, more reliable. The scan only needs to return item_type + brand + model.

Safety and content guardrails

What AI should never generate

Tasks involving gas, electrical, or structural work without "hire a professional" guidance.
Medical device maintenance (health category items should always use curated templates).
Tasks that could void a warranty without mentioning the risk.
Specific product recommendations with affiliate links (affiliate links are added by the nudgii system, never by the AI).
Legal advice about regulations (the AI can mention "check local requirements" but never claim something is legally required unless it's in the curated template).

Content review queue

All AI-generated "Why it matters" and "How to do it" content is stored. A weekly automated scan flags any templates containing safety-sensitive keywords (gas, electrical, wiring, plumbing, asbestos, mould). Flagged templates get human review before being promoted to curated.
